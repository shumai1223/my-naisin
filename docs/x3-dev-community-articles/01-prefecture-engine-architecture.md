## 本文ドラフト

### タイトル案: 47都道府県バラバラの入試制度を、1つの計算エンジンで扱う設計にした話

中学生向けの内申点・偏差値計算サイト「My Naishin」を個人で開発・運営しています。このサイトの
核心は、日本の公立高校入試における「内申点」の算出方法が**47都道府県すべてバラバラ**という
事実を、1つの型安全なTypeScriptのデータ構造でどう表現するか、という設計問題でした。

#### 問題: 制度のバリエーションが多すぎる

内申点の算出方法は、都道府県の教育委員会がそれぞれ独自に定めています。バラバラな軸を挙げると：

- **対象学年**: 中3のみ／中2〜中3／中1〜中3のいずれか
- **学年別の倍率**: 均等か、中3を重く傾斜させるか（比率も1:1:2〜1:2:3など様々）
- **実技教科の倍率**: 主要5教科と同等か、最大10倍（鹿児島県）まで幅がある
- **満点**: 45点〜660点まで、学年数×倍率の組み合わせでバラバラ

さらに厄介なのは、内申点が決まった**その先**——学力検査との合算方法——も県ごとに全く違う制度に
なっていることです。神奈川県は志望校が選ぶ6段階の比率で1000点満点のS値に変換、大阪府は5つの
「タイプ」から高校が選ぶ、千葉県は0.5〜2の連続的な係数（K値）を高校が独自に設定、東京都は
英語スピーキングテスト（ESAT-J）という学力検査に含まれない要素まで加わります。

#### 解決策1: 内申点そのものは1つの設定オブジェクトで表現できた

内申点自体の算出ロジックは、実はかなりの部分が「対象学年・学年別倍率・主要教科倍率・実技教科倍率・
満点」という共通のパラメータで表現できることが分かりました。

```typescript
interface PrefectureConfig {
  code: string;
  targetGrades: number[];              // 例: [2, 3]
  gradeMultipliers: Record<number, number>; // 例: { 1: 0, 2: 1, 3: 2 }
  coreMultiplier: number;               // 主要5教科の倍率
  practicalMultiplier: number;          // 実技4教科の倍率
  maxScore: number;                     // 満点
  sourceUrl: string;                    // 教育委員会の一次情報URL
  // ...
}
```

47都道府県ぶんのこの設定値を配列で持ち、計算ロジック自体は1つの純関数にする。新しい県を
追加する（あるいは制度改定に対応する）のは、基本的にこの設定オブジェクトを1つ足すだけで
済みます。ロジックの分岐を都道府県の数だけ書かずに済んだのは、この抽象化が効いたポイントでした。

#### 解決策2: 「学力検査との合算」は無理に一般化しなかった

内申点の先にある総合得点の計算は、最初は同じように1つの汎用エンジンで表現しようとしました。
実際、47都道府県のうち兵庫・京都・栃木・新潟・鳥取の5県は、「内申・学力をそれぞれ換算満点へ
線形換算し、選んだ比率オプションのもとで合算する」という共通モデル（`engine.ts`）でうまく
表現できました。固定比率の県も、選択式比率の県も、「比率オプションの違い」として同じ関数で
扱えたのです。

```typescript
// 共通エンジン（5県で共用）
function computeReportRaw(
  report: { coreMultiplier: number; practicalMultiplier: number },
  evals: { coreSum: number; practicalSum: number },
): number {
  return evals.coreSum * report.coreMultiplier + evals.practicalSum * report.practicalMultiplier;
}
```

ところが、東京（ESAT-J）・神奈川（S1/S2値・6段階比率）・大阪（5タイプ選択制）・愛知（評価方法
Ⅰ〜Ⅴ）・千葉（連続係数K）・埼玉・福岡・北海道の8都道府県は、この共通モデルに無理やり押し込もうと
すると条件分岐だらけになり、かえって可読性もテストしやすさも落ちることが分かりました。最終的に、
この8県は**個別のファイルとして正直に分離**しました（`kanagawa.ts`・`osaka.ts`・`aichi.ts`...）。

各ファイルの冒頭には必ず「なぜ共通エンジンの対象外なのか」を1行でコメントしています。

```typescript
// 統一エンジン（engine.ts/registry.ts）の対象外＝神奈川県は内申点(135点満点)と
// 学力検査点(500点満点)を、志望校ごとに異なる比率(4:6〜7:3等)で100点満点換算し
// 合算するS1値、特色検査を加えたS2値という独自方式のため個別実装している。
```

**「共通化できるものは共通化し、無理に共通化しようとしたものは正直に特殊化して、その境界を
コメントで明示する」**——この設計判断が、47都道府県という一見カオスなドメインを破綻なく
コードに落とし込めた一番の理由だったと思います。

#### テストで「手抜き」を機械的に検知する

個人開発で品質を保つ工夫として、通常のユニットテストに加えて次のようなテストを書いています：

- **`uniqueness.test.ts`**: 都道府県別ページの本文が、県名を差し替えただけの金太郎飴コンテンツに
  なっていないかを検知する（量産に走った時の自己検知）
- **`internal-link-graph.test.ts`**: 全静的ページが被リンク数2件以上を持つか（登録されているだけで
  実際にはどこからもリンクされていない「孤児ページ」を防ぐ）
- **`rich-results-*.test.ts`**: FAQPage/HowTo/BreadcrumbList/Datasetの構造化データが全ページで
  抜け漏れなく実装されているか（例外は理由付きでホワイトリスト化し、審査なしの抜け道を防ぐ）

これらは「機能が動くか」ではなく「量が増えても質が落ちていないか」を機械的にチェックするための
テストで、1人で数十〜100ページ規模のサイトを保守する上で欠かせない仕組みになっています。

#### この計算エンジンをAPI/MCPとしても公開している

作った計算ロジックは、REST API・OpenAPI 3.1仕様・MCP（Model Context Protocol）サーバーとして
無料公開しています。MCPサーバーは `list_prefectures` / `get_prefecture` / `calculate_naishin` /
`compare_prefectures` / `reverse_calc` / `target_to_required_grades` / `build_study_plan` の
7ツールを提供し、AIエージェントが直接この計算エンジンを呼び出せるようにしています
（詳細: [https://my-naishin.com/developers](https://my-naishin.com/developers)）。

47都道府県という「ドメインの複雑さ」を型と純関数でどう表現するかは、教育系に限らず地域差分の
ある制度・料金体系を扱うプロダクトなら似た課題に直面すると思います。何かの参考になれば幸いです。
