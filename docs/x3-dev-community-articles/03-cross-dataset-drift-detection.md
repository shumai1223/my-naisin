## 本文ドラフト

### タイトル案: データベースを使わないサイトで、2つの別データセットの整合性をどう守るか

「内申点・偏差値計算サイト」を個人開発していると、47都道府県ぶんのデータをいくつかの
TypeScriptファイルに分けて管理することになります。My Naishinには大きく2つのデータ
セットがあります。

- `prefectures.ts`: 47都道府県すべての「内申点そのもの」の計算方法（正準データ）
- `total-score/registry.ts` + `explainers.ts`: 一部の都道府県だけ対応している
  「総合得点（内申点＋学力検査）」の計算・解説データ

この2つは別ファイル・別配列として管理しています。理由は単純で、総合得点の計算は
内申点よりずっと複雑（学校ごとの比率選択・独自の変換式等）で、全県対応がまだできて
いないためです。しかし「別ファイルで管理する」ということは、**両者がこっそり
食い違っていく（ドリフトする）リスク**を抱えるということでもあります。

#### よくあるデータ不整合のパターン

- `total-score`側にしか存在しない、実在しない都道府県コードが紛れ込む
- 同じ都道府県が「計算機」と「解説」の両方に二重登録されてしまう
- 県名の表記が2つのファイルで食い違う（例: 片方は「兵庫県」、もう片方は「兵庫」）

RDBならこれは外部キー制約やユニーク制約で機械的に防げますが、静的サイトの
TypeScript配列にはそういった仕組みはありません。そこで、**この不変条件そのものを
jestのテストとしてCIに固定する**ことにしました。

```typescript
const VALID_CODES = new Set(PREFECTURES.map((p) => p.code));

test('計算機の県コードはすべて実在する47県を指す', () => {
  for (const code of VERIFIED_TOTAL_SCORE_CODES) {
    expect(VALID_CODES.has(code)).toBe(true);
  }
});

test('計算機と解説は排他（同一県を二重管理しない）', () => {
  const calc = new Set(VERIFIED_TOTAL_SCORE_CODES);
  for (const code of EXPLAINER_CODES) {
    expect(calc.has(code)).toBe(false);
  }
});

test('計算機の県名は PREFECTURES の県名と一致（ドリフト検出）', () => {
  for (const code of VERIFIED_TOTAL_SCORE_CODES) {
    expect(getTotalScoreSystem(code)?.name).toBe(getPrefectureByCode(code)?.name);
  }
});
```

#### この設計の効いているところ

このテストは実装コードを一切変更せず、**既存の2つのデータ配列を読むだけ**で書けます。
新しい県を`total-score`に追加するたびに、このテストが自動的にその県の整合性
（実在するか・二重登録していないか・県名が一致するか）を検証してくれるため、
「新しい県を追加したら手動でチェックリストを確認する」という運用に頼らずに済みます。

DBを使わない構成でも、「複数の関連データが食い違わない」という不変条件自体を
コードとして書いておけば、レビューで見逃しても最終防衛ラインとしてCIが必ず
検出してくれる、というのがこのパターンの要点です。
