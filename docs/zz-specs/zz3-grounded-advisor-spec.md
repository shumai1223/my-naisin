# ZZ-3 グラウンデッド・アドバイザー 設計仕様書（Fable 5設計・2026-07-23）

ZZ-3a/3b/3c実装時の**準拠必須**仕様。Sonnet loopはこの仕様から逸脱する設計判断をしない（変更したい場合はloop-question-noteに理由を書いて👤裁定待ち）。

## 0. 核心原則（これだけは絶対）

**回答に含まれる全ての数値は、決定論エンジンの出力台帳（grounding ledger）由来でなければならない。** AIどころかテンプレート自身も数値を「作文」してはならない。検証器はこれを機械的に強制する。

## 1. アーキテクチャ（3層）

```
[質問] → ①分類器(決定論) → ②エンジン呼出+台帳構築 → ③レンダラ(テンプレ) → ④検証器 → [回答]
```

- **①分類器**: キーワード/正規表現ベースの決定論ルーティング（AI不使用）。対応質問タイプ: 内申点計算/目標逆算(reverse)/県間比較/総合得点/制度説明/範囲外。**範囲外は「答えられない」と正直に返す**（無理に答えない）。
- **②台帳（GroundingLedger）**: エンジン呼出の結果を `{ value: string(表示形), source: 'naishin-engine'|'total-score'|'prefectures'|..., context: prefCode }` の配列として蓄積。**数値のフォーマット（丸め・単位付与）はエンジン→台帳登録時に一度だけ行い、台帳には表示形そのものを格納する**（レンダラ側で再丸め・再計算を禁止。「約3.7」を出したければ台帳に「約3.7」を登録する）。
- **③レンダラ**: テンプレート文字列の数値スロットは台帳エントリの参照のみ。テンプレ地の文に算用数字を書くことを原則禁止（例外は§3の構造定数のみ）。
- **④検証器（verifyGrounding）**: 純関数 `(renderedText, ledger, prefContext) => { ok, violations[] }`。**テスト時とランタイムの両方で同じ関数を使う**。ランタイムでfailした場合は該当回答を破棄し、決定論の安全定型文へフォールバック（エラーを見せない）。

## 2. 検証器の数値抽出仕様

1. 前処理: markdownリンクは**href部分を除去しリンクテキストのみ検証対象**（URL内の数字 /report/2026 等を誤検出しない）。コードブロックは出力自体を禁止（advisorはコードを出さない）。
2. 全角数字→半角に正規化してから抽出。
3. 抽出パターン: `[0-9]+([.,][0-9]+)?` に加え、**漢数字の常用パターン**（一〜九＋「年生/学期/教科/段階」の直結形、および「三百」等の位取り漢数字）も抽出する。回避経路（漢数字で書けば通る）を残さない。
4. 範囲表現 `300〜350` は両端を個別数値として検証。
5. マッチ判定: 抽出トークンが (a)台帳の表示形 (b)§3構造定数 のいずれかと**完全一致**しなければ violation。部分一致・近似一致は認めない。

## 3. 構造定数ホワイトリスト（唯一の台帳外数値）

`ADVISOR_STRUCTURAL_CONSTANTS` として1モジュールに集約: 「9教科」「5教科」「5段階」「47都道府県」「1〜3年生」「1〜3学期」および年度表記（2026/2027年度＝config由来）。**これ以外の定数追加はテスト変更を伴う=差分レビューで可視化される**構造にする。満点・倍率・評定値は構造定数ではなくエンジン由来（県ごとに違うため）。

## 4. 県コンテキスト混線ガード

v1は**1回答=1県**制約とする（比較質問は「県Aの回答」+「県Bの回答」の2ブロック構成で、各ブロックに独立した台帳を持つ）。台帳エントリはprefCode必須・レンダラはブロックのprefCodeと台帳エントリのprefCodeの一致をassert。東京の1020点が大阪の回答に混入する事故を構造的に排除する。

## 5. 断定進路指導の禁止（judgment lint）

数値検証と独立に、禁止フレーズlintを掛ける: 「受かります/合格できます/合格です/安全です/大丈夫です/確実/絶対」等の断定形（推量形「〜の可能性があります」はエンジン由来の判定値がある場合のみ可）。全回答末尾に定型「最終的な判断は学校の先生・保護者の方と相談してください」を必須とし、その存在もテストでassert。

## 6. 敵対的テスト50本の構成（ZZ-3b DoD）

| カテゴリ | 本数 | 例 |
|---|---|---|
| 直接捏造（台帳外数値をテンプレに混入させた変異体がfailすること） | 10 | 地の文に「約320点」直書き→fail |
| 丸めドリフト | 6 | 台帳3.67に対し本文3.7→fail（台帳に3.7を登録した場合のみpass） |
| 漢数字回避 | 6 | 「三百点」→抽出されfail |
| 全角数字回避 | 4 | 「３２０点」→正規化されfail |
| 県混線 | 8 | 大阪ブロックに東京満点1020→fail |
| 単位すり替え | 4 | 「65%」vs「65点」（台帳は表示形一致なので単位違いはfail） |
| エンジンエラー経路 | 6 | エンジンがエラー時、回答に構造定数以外の数値ゼロ+安全定型文 |
| 断定lint | 6 | 「この内申なら受かります」→fail |

**mutation-testの発想**: 検証器が「正しい実装をpassさせる」テストだけでなく、「壊した実装を確実にfailさせる」テストを必ず対で書く（検証器自体が穴だと全部が無意味になるため、こちらが本体）。
