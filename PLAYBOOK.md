# 横展開プレイブック（my-naishin の勝ち筋を再現するための型）

> このドキュメントは「my-naishin.com を本業レベルへ育てる」ための**再現可能な仕組み**を言語化したもの。
> 守る資産は1つのサイトではなく、**この型（プレイブック）と運用ループ**。次のサイトはこれをコピーして立ち上げる。

---

## 0. なぜ横展開なのか（前提）

- 単一ニッチ（中学生×内申点）は**検索総量に天井**があり、**学生オーディエンスは広告単価が安い**。
- → 本業（月30万円〜）への道は **(A) 1サイトの単価を上げる（広告→リード送客）** ＋ **(B) 同じ型を高単価ニッチへ横展開**。
- my-naishin で確立した「**計算ツール × セグメント別プログラマティック × E-E-A-T × GSC駆動改善**」は、そのまま他ニッチに移植できる。

---

## 1. 勝ちパターンの構造（4点セット）

このサイトが伸びている理由を分解すると、再現すべきは次の4つ。

### ① コア計算ツール（滞在・指名・被リンクの起点）
- ユーザーが「自分の数字」を入れて即答が返るツール（内申点・偏差値・評定平均）。
- 結果はローカル保存、入力は最小。**1意図1ツール**（汎用ページより専用ページがCTR 20〜30%と圧勝：`/hyotei-heikin` が実例）。

### ② セグメント別プログラマティック生成（ロングテール網羅）
- 1つの軸（このサイトでは**47都道府県**）でページを自動量産。`src/lib/prefectures.ts` のデータ → `[prefecture]/naishin` で全県ページ。
- 各ページに**固有の一次データ**（満点・対象学年・倍率・出典URL）を持たせ、薄いページにしない。
- 軸は何でもよい：都道府県／年収帯／資格区分／大学・学部 など「ユーザーが自分ごと化できる切り口」。

### ③ E-E-A-T（個人サイトが上位を取れる理由）
- 運営者プロフィール（`/about/editor-profile`）、`/quality`、出典明記、最終検証日、文科省一次情報の引用。
- 構造化データ：`FAQPage` / `HowTo` / `BreadcrumbList` / `WebApplication`（`src/components/StructuredData/`）。
- 一次情報リンクの**死活監視を自動化**（`link-monitor.yml`）＝情報の鮮度を仕組みで担保。

### ④ 収益化：複合エンジン（アフィリ単体では届かない・実測で判明）
- **AdSenseは撤退済み**（2026-07-04）。理由は広告収益の天井以前に、審査対応で入れていた`auditHide`が保護者向けリードCTAを全ページで非表示にする自傷事故を起こしていたと判明したため。撤去した瞬間に休眠していたCTAが全面解凍した＝**広告とリード送客は同じ画面上で共食いする**という実測の教訓。
- **アフィリエイト単体では収益目標に構造的に届かない**（必要送客クリック数に対し現実の到達クリック数が約10倍不足という実測に基づく試算）。**支配変数は実CVR**で、確定発生は長期間ゼロだった（生徒起点のクリックは事実上発生せず、保護者接点がボトルネック）。
  - 実装：`src/components/ParentLeadCTA.tsx`（保護者向け。子ども向け煽りCTAと分離）＋`src/lib/ev-engine.ts`（EV計算の単一ソース）＋`src/lib/lead-config-engine.ts`（面ごとのオファー解決）。
- **複合エンジンが必須という結論**：①アフィリフル稼働（EVランキングで機械的に上位オファーへ誘導）②特単（発生量が閾値を超えたら交渉発動）③直接契約/掲載枠（`src/lib/sponsor-slots.ts`）④API/MCPのB2B課金（`src/lib/api-tiers.ts`・Pro/Scale）。単一チャネル依存を避ける設計そのものが「型」。
- **真のボトルネックは保護者接点（C_p）**：生徒がツールを使っても保護者にリーチしなければ収益化しない。QR共有・面談パック・LINE即時報酬など「その場で親に見せる」導線（`ParentWindowBridge`・`ParentShareLinkButton`）を結果画面の直後に置くのが型。

## 2. 運用ループ（張り付かないための自動化）

「世話」をやめ、「向こうから改善点が来る」状態にする。

| 仕組み | 役割 | 実体 |
|---|---|---|
| GSC週次レポート | 毎週「次に打つ手」をIssueで通知 | `gsc-weekly.yml` + `gsc-weekly-report.ts`（striking distance / 高imp低CTR / 順位下落 / 急上昇 / 新規） |
| 新面クエリマイニング | 表示回数はあるが専用面が無いクエリを機械抽出→決定論的に答えられるものだけ面化 | `src/lib/query-mining.ts`（同レポートに統合） |
| CTR自動改善ループ | 順位帯別の期待CTRを大きく下回る面を検出→title/meta改善→3週後に効果測定 | `src/lib/ctr-improvement.ts` + `ctr-improvement-log.ts`（同レポートに統合。ガード：流入上位5面は対象外・同一面の変更は3週間隔） |
| 新面インデックス監視 | 新設バッチが発見(index)されているかを週次で確認 | `src/lib/new-page-launch-tracker.ts`（同レポートに統合） |
| A/B実験の弾倉 | CTAコピー/配置/オファー順のバリアントをキューに積んでおき常時2本以上を回転 | `src/lib/experiments.ts`（`queued`ステータス・`nextQueuedExperiment()`） |
| 内部リンクグラフ監査 | 準孤児（クリックで辿れないページ）を検知 | `src/lib/internal-link-graph.ts` + 回帰テスト（被リンク1件超を強制） |
| 一次情報リンク監視 | 出典切れを自動検知→Issue | `link-monitor.yml` |
| プログラマティック生成 | 手作業なしでページ量産 | データ駆動の `[segment]` ルート |
| 被リンクエンジン | 受動的に被リンク獲得 | 埋め込みウィジェット `/embed/*` |

**月次でやること（30〜60分）**: GSCレポートのstriking distance上位3〜5件に内部リンク／見出し最適化を当てる。それだけ。

**実測で効いたと確認できているパターン（2026-07-11時点）**:
- **curiosity-gap型タイトル**：「中1は入らない！中2＋中3×2倍」のように、その都道府県固有の制度の意外性を数値付きでタイトル・冒頭に出す（`[prefecture]/naishin/page.tsx`の`PREFECTURE_META_OVERRIDES`）。kanagawa等の先行適用面はCTR自動改善ループの劣後リストから外れており、汎用テンプレより高いCTRで安定している。
- **専用ページ＞汎用ページ**：1意図1URLの専用計算機（例：`/hyotei-heikin`）は、同種の情報を汎用ハブに同居させるより高いCTRを継続的に記録している。
- **結果直後の保護者導線**：計算結果が出た直後（不安が最大化する瞬間）にQR共有・面談パック等の保護者向けCTAを置く設計思想は変えていないが、実クリック数自体はまだ乏しく「効いた」と断定できる段階ではない（母数不足・反証ゲート思想により拙速な結論を避ける）。

## 3. 新サイト立ち上げチェックリスト

```
[ ] ニッチ選定（§4の基準でスコアリング）
[ ] セグメント軸を決める（都道府県/年収/資格/大学 など）とデータ表を作る（=堀）
[ ] コア計算ツールを1つ実装（1意図1ツール）
[ ] セグメント別ページを自動生成（各ページに固有データ＋出典）
[ ] E-E-A-T：運営者プロフィール / 品質ページ / 出典 / 構造化データ
[ ] 収益導線：リード送客CTA（本命）を結果直後に。AdSenseは同時併用しない（§1-④の自傷事故を参照）
[ ] sitemap / robots / canonical / OG：page-registry.ts（単一ソース）→sitemap.tsを自動生成する構成にする（手動配列は新規ルート登録漏れの前科あり）
[ ] 品質ゲートをCIに接続：uniquenessテスト（プログラマティック生成面のscaled-content対策）・a11y監査（フォームのラベル/コントラスト）・rich-results監査（FAQ/HowTo/Dataset/Breadcrumbの網羅）・internal-link-graph監査（準孤児検知）
[ ] GSC・サーチコンソール登録、週次レポートのcronを接続
[ ] 一次情報リンク監視を接続
[ ] 被リンク用の埋め込みウィジェット
```

技術スタックはこのリポジトリを雛形に：**Next.js(App Router) + Tailwind + OpenNext/Cloudflare + tsx scripts + GitHub Actions**。

**コピー元＝「テンプレコピー」方式（共有npmパッケージにはしない）**：my-shingaku立ち上げ時にF-7で確立した方式。my-naishinのロジックを直接複製せず、汎用部分だけを`src/lib/ev-engine.ts`（EV計算式）・`src/lib/lead-config-engine.ts`（オファー解決パターン）に分離し、新サイトはこの2ファイルをコピーして自サイト固有のデータ（オファー一覧・県別オーバーライド等）だけを書く。`page-registry.ts`（sitemap自動生成）・`track.ts`（計装イベント名）・`seasonal.ts`（季節オファー切替）も同じ思想でコピー元にする。詳細設計は[[session-2026-07-09-f7-playbook-extraction-design]]。

## 4. 高単価ニッチ選定（次の1サイト）

> **2026-07-11更新：第1候補は実行済み**。下記の「結論」で示した大学受験×学費・一人暮らしニッチは、**my-shingaku.com として2026-07-10に本番デプロイ済み**（ドメイン取得・GSC/Bing登録・my-naishinの品質スタック(page-registry/回帰テスト/uniquenessゲート)を移植済み）。以下§4は選定当時の記録として残し、次サイト（第3サイト）検討時は[[session-2026-07-09-third-playbook-research]]を参照：英検が次点候補だが先行競合多数で単純複製は非推奨、中学受験・資格系は「検証可能な公的一次情報が存在しない」という構造的理由でmy-naishin型（決定論エンジン×検証済み一次データ）が機能せず除外済み。次の1サイトは慎重な再選定が必要。

### 選定基準（各5点・合計で評価）
1. **RPM/単価**（広告＋リード単価）
2. **プログラマティック適性**（自然なセグメント軸があるか）
3. **計算ツール適性**（"自分の数字"を入れる動機があるか）
4. **競合の隙**（個人のE-E-A-Tで割って入れるか）
5. **運営者適性**（しゅうまいの土地勘・信頼性・法的リスクの低さ）

### 推奨ランキング

#### 🥇 第1候補：高校生／大学受験 ×「学費・奨学金・一人暮らし費用」シミュレーター
- **これが本命**。理由：**今の中学生ユーザーがそのまま育つ**隣接領域で、土地勘・E-E-A-Tを引き継げる。
- セグメント軸：大学／学部／地域（=地方からの下宿）。計算ツール：4年間総額・奨学金返済・一人暮らしの初期費用＆毎月。
- 収益が段違い：**予備校・通信教育の資料請求＋一人暮らしの引越し一括見積もり（高単価）＋奨学金/銀行**。決裁者は**保護者＝お金を出す層**。
- スコア: 単価4 / プログラマティック5 / ツール5 / 競合3 / 適性5 = **22/25**

#### 🥈 第2候補：税金・手取り計算（年収帯 × 都道府県の住民税／副業・フリーランス）
- 大人・お金の領域で**RPMが最も高い**。転職エージェント・会計ソフト（freee/マネフォ）・ふるさと納税の送客。
- 弱点：YMYL寄りで個人の信頼性ハードルが高い、競合が強い、土地勘が薄い。
- スコア: 単価5 / プログラマティック5 / ツール5 / 競合2 / 適性2 = **19/25**

#### 🥉 第3候補：引越し・賃貸初期費用（都道府県・市区別の相場＆計算）
- 47県プログラマティックを**そのまま流用**でき、**一括見積もりのリード単価が非常に高い**。
- 弱点：競合（大手比較サイト）が極めて強い、個人の権威で勝ちにくい。
- スコア: 単価5 / プログラマティック5 / ツール4 / 競合1 / 適性2 = **17/25**

### 結論
**第1候補（大学受験×学費・一人暮らし）から始める。** 既存サイトの読者が進学とともに流れ込み、土地勘と信頼性を引き継いだまま、収益単価だけが跳ね上がる。受験が落ち着いた後、第2候補（税金）でRPMの高い大人市場へ。

> ⚠️ 進める順序：**受験が最優先**。横展開は「受験後にこの型をコピーして1サイト」で十分。今は my-naishin の自動化（§2）と保護者導線（§1-④）を仕込んで"放牧"する。
